# 面试准备速记（aiStudy 项目）

## 1) 先用一句话介绍项目

这是一个“以单篇论文为中心的学习系统”：把 PDF 论文抽象成 `Asset`，围绕它提供阅读、检索增强问答（RAG）、思维导图和锚点笔记，目标是形成“阅读 -> 理解 -> 提炼 -> 复习”的闭环。

## 2) 技术栈为什么这样选

- **前端：Vue 3 + TS + Vite**
  - Vue 3 组件化成本低，适合快速搭建学习工作区；TS 让 API 契约更稳定；Vite 提供快反馈。
- **后端：FastAPI + Pydantic**
  - FastAPI 对异步 IO、Schema 校验、OpenAPI 友好，适合“上传/解析/轮询/检索/问答”这类接口密集场景。
- **异步任务：Celery + Redis**
  - PDF 解析、向量化、资源生成都属于长任务，放到 Celery 避免阻塞请求线程。
- **数据层：PostgreSQL + pgvector**
  - 元数据、状态流转、业务关系都在 PG；向量检索同库完成，降低早期系统复杂度。
- **对象存储：阿里云 OSS**
  - MinerU 解析要求公网可访问 PDF，先上传 OSS 再提交解析任务。

## 3) 主链路（面试高频）

1. 用户上传 PDF。
2. 后端保存资产和文件记录，PDF 上传 OSS。
3. 异步触发 MinerU 解析并轮询结果。
4. 解析产物规范化为统一 `parsed_json`。
5. 基于 `parsed_json` 生成 `document_chunks`。
6. 调用 embedding 模型写入 pgvector。
7. 问答阶段：先检索，再由 LLM 生成回答，并返回 citation（可回跳页码/块）。

## 4) 你可以这样解释 RAG 价值

- 没有 RAG：模型容易“看不到原文依据”，回答不可追溯。
- 有 RAG：先做资产内召回，再让模型基于证据作答，前端展示引用并支持回跳。
- 这个项目强调**单资产边界**，先保证准确性和可解释性，再考虑跨资产检索。

## 5) 关键数据对象（建议背下来）

- `assets`：论文资产主对象。
- `asset_files`：上传原始文件与存储位置。
- `document_parses`：解析任务与状态。
- `document_chunks`：切分后的检索单元（含 embedding）。
- `chat_sessions / chat_messages / citations`：问答会话、消息、引用。
- `anchors / notes`：锚点与笔记（定位语义和内容分离）。

## 6) 面试官追加 6 问（可直接背）

### Q1：为什么选择 FastAPI + Celery？Redis 在项目中起什么作用？

**标准答法：**

- `FastAPI` 负责在线请求：上传、状态查询、检索、问答接口，特点是开发快、类型校验强、文档自动化友好。
- `Celery` 负责离线/长任务：MinerU 解析、chunk 构建、向量化、重试任务，避免把几十秒甚至分钟级任务塞进 HTTP 请求。
- `Redis` 在这里主要是 **Celery 的 Broker（消息队列）和可选 Result Backend（结果存储）**：
  - Broker：把“需要异步执行的任务”从 Web 进程转发给 Worker；
  - Backend：记录任务执行状态，便于轮询与观测；
  - 同时 Redis 也可承接轻量缓存（如短期状态），但在本项目里核心价值是任务编排基础设施。

一句话总结：**FastAPI 管在线，Celery 管离线，Redis 负责把两者解耦并串起来。**

### Q2：`parsed_json` 是不是用 Python 规则处理 markdown？可扩展性如何？中文论文怎么办？

**标准答法：**

- 不是“只靠 markdown 正则切一刀”。当前链路是：MinerU 返回结构化中间结果（如 `content_list.json`、`middle.json`、markdown），后端做**规范化映射**，统一落成平台内部 `parsed_json`。
- `parsed_json` 的定位是“平台中间语义层”，不是单一供应商输出的镜像。这样做是为了：
  1. 前端阅读器、检索、导图都依赖同一结构；
  2. 后续更换解析器（或多解析器）时，上层业务基本不动。
- 可扩展性做法：
  - 把“解析结果 -> 统一结构”的规则做成可演进的 normalizer；
  - 尽量依赖版式/结构字段（章节、块、页码）而非语言本身；
  - 针对中文论文差异（标题层级、标点、分段习惯）可增加语言/模板分支规则，但输出契约不变。

一句话总结：**我们不是绑定 markdown，而是绑定统一 schema；语言变化是 normalizer 规则问题，不是上层架构问题。**

### Q3：RAG 链路怎么实现？`document_chunk` 是什么？为什么不用 markdown 直接分块？粒度怎么保证？

**标准答法：**

- RAG 原理：先检索，再生成。模型回答前先拿到与问题相关的文档证据，降低幻觉并提升可追溯性。
- 本项目链路：
  1. `parsed_json` 结构化文本；
  2. 按规则切成 `document_chunks`；
  3. chunk 向量化写入 pgvector；
  4. 问答时向量检索召回 topK；
  5. LLM 基于召回证据生成回答 + citation。
- `document_chunk` 可以理解为“检索最小语义单元”，通常包含：文本内容、页码、段落号、block_ids、章节路径、embedding。
- 为什么不直接 markdown 分块：
  - markdown 更偏展示格式，定位信息（页码、块 id、段落映射）不稳定；
  - 直接 markdown 分块会弱化“回跳定位”和“引用追踪”；
  - `parsed_json` 保留更多结构语义，便于稳定切分与后续功能复用。
- 粒度控制：
  - 以章节边界 + 长度阈值双约束切分；
  - 保留来源 block_ids，避免 chunk 过粗导致噪声，过细导致语义碎片化；
  - 通过离线评估（命中率、引用可用率）迭代阈值。

一句话总结：**chunk 是“既能检索又能回跳”的最小证据单元，不是随便切的文本片段。**

### Q4：数据库为什么选 PostgreSQL + pgvector？好处是什么？

**标准答法：**

- MVP 阶段优先稳定交付，`PostgreSQL + pgvector` 的优势是：
  1. 业务数据和向量数据同库，开发和运维复杂度低；
  2. 一套事务/备份/权限体系即可；
  3. 早期数据量下性能和可维护性平衡较好。
- 相比“单独再上一个向量数据库”，这个组合更适合当前阶段快速迭代。
- 当规模上来（更大语料、更高 QPS、更复杂 ANN 策略）再考虑拆分专用向量引擎，是更稳妥的演进路径。

一句话总结：**先用同库方案把链路做稳，再按规模驱动拆分。**

### Q5：citation 是什么？RAG 策略是什么？怎么保证检索准确？为什么没选其他策略？

**标准答法：**

- `citation` 是回答中的“证据引用”，至少要能回溯到页码/段落/block/section，让用户知道“这句话来自哪里”。
- 当前策略（MVP）：**单资产向量召回 + 结构化引用返回 + 回答边界约束**。
  - 单资产：降低语义漂移；
  - topK 召回：控制上下文成本；
  - 回答提示词约束：仅基于证据作答，不足就说不知道；
  - 输出强制 citation：可验证、可回跳。
- 准确性保障：
  1. 数据侧：chunk 质量和粒度可控；
  2. 检索侧：统一 embedding 模型、合理 topK；
  3. 生成侧：拒答策略 + 引用必填；
  4. 产品侧：前端展示引用，支持人工复核。
- 常见但暂未采用的 RAG 策略：
  - 混合检索（BM25 + 向量）；
  - rerank 重排（cross-encoder）；
  - 多路召回/查询重写；
  - Knowledge Graph RAG；
  - 多跳检索与 Agent 工具调用。
- 为什么当前不选：
  - 这些策略效果更强但系统复杂度、延迟、成本都会上升；
  - 当前优先目标是“单资产可追溯回答”主链路稳定，后续再迭代增强。

一句话总结：**先把“可追溯、可回跳、可验证”做扎实，再上复杂召回策略。**

### Q6：锚点（Anchor）是什么？为什么要设计锚点？

**标准答法：**

- 锚点是对论文中某个知识位置的统一定位对象，比如 `page_no + block_id (+ paragraph_no)`，或导图节点定位。
- 设计锚点的原因：
  1. 统一“阅读器选区、问答引用、导图节点、笔记”这四类能力的定位语义；
  2. 支持从任何入口回跳原文；
  3. 锚点和笔记分离后，可一个锚点挂多条笔记，且后续升级定位算法不破坏历史内容。
- 没有锚点会出现的问题：引用无法稳定回跳、笔记和问答定位不一致、跨模块联动成本高。

一句话总结：**锚点是全平台“知识定位坐标系”，是联动能力的基础。**

## 7) 你可以主动说的“工程化意识”

- 长链路都设计了状态机（解析状态、知识库构建状态、问答会话）。
- 关键阶段可重试（例如 parse retry、chunks rebuild）。
- API 契约尽量结构化（引用、锚点、检索命中均可复用）。
- 按 Spec 迭代交付，先 MVP 再增强，降低返工风险。

## 9) 你这轮新增的 5 个高频追问（进阶版）

### Q7：阅读器如何实现“回跳定位”？

**标准答法：**

- 回跳依赖统一锚点对象（Anchor），核心字段通常是：`page_no + block_id (+ paragraph_no)`。
- 前端回跳流程：
  1. 用户在问答 citation / 笔记 / 导图节点点击“查看原文”；
  2. 前端拿到锚点并先跳页（页级定位，保证总能落在正确页面）；
  3. 在当前页文本层或块列表中用 `block_id` 高亮对应段落；
  4. 若文本层映射不稳定，降级为“页级定位 + 相邻块人工确认”。
- 后端职责：保证 citation、note、mindmap 节点都返回同一套可定位字段，避免不同模块各说各话。
- 设计重点：先保证“能回到正确页面”，再逐步提高到“字符级精确定位”。

一句话总结：**回跳是“统一锚点 + 页级兜底 + 块级高亮”的组合策略。**

### Q8：实现过程中遇到过什么困难？你是怎么解决的？

**标准答法：**

你可以按“问题 -> 影响 -> 处理 -> 结果”讲，下面是可复述模板：

1. **外部依赖波动（OSS / MinerU / 模型接口）**
   - 影响：主链路容易因第三方抖动失败。
   - 处理：异步任务化 + 状态轮询 + 可重试接口（parse retry / rebuild）。
   - 结果：失败可恢复，用户感知从“卡死”变成“可观测、可重试”。

2. **解析输出不完全稳定**
   - 影响：直接消费原始输出会导致上层功能被动改动。
   - 处理：引入 `parsed_json` 作为内部统一语义层，做容错 normalizer。
   - 结果：阅读器、检索、导图共享同一输入契约，演进成本下降。

3. **chunk 粒度难平衡**
   - 影响：过粗召回噪声大，过细语义断裂。
   - 处理：章节边界 + 长度阈值双约束，并保留 block_ids 追踪来源。
   - 结果：检索命中和回跳可用性可同时兼顾，并可持续调参。

### Q9：面试官常问“如果项目出现 X 问题你怎么办？”（发散题库）

**可准备的场景题 + 答题框架：**

1. **如果检索准确率突然下降怎么办？**
   - 先看数据（chunk 是否异常、embedding 是否变更）；
   - 再看检索参数（topK、阈值）；
   - 最后看生成提示词是否放宽过度；
   - 回滚最近参数并做离线评估对比。

2. **如果模型成本过高怎么办？**
   - 分层降本：先控召回上下文长度、再切更轻模型、最后做缓存与请求合并。

3. **如果解析队列积压严重怎么办？**
   - 提升 Worker 并发，拆分队列优先级（解析/向量化分队列），对超时任务做隔离重试。

4. **如果用户说 citation 对不上原文怎么办？**
   - 检查锚点映射链路（chunk -> citation -> anchor -> reader）；
   - 增加引用落库校验和前端回跳自检；
   - 短期先页级兜底，长期补字符级定位。

5. **如果要支持多用户权限隔离怎么办？**
   - 所有核心对象引入 `user_id` 强过滤；
   - API 层做鉴权中间件；
   - 检索和问答请求默认按用户作用域 + asset 范围双重限制。

**通用作答模板：**“先止血（降级/回滚）-> 再定位（日志/指标/样本复现）-> 再修复（小步上线）-> 最后复盘（加监控和防回归）”。

### Q10：你的创新点在哪里？对标产品有哪些？你的优势是什么？

**标准答法：**

- 同类产品大致有三类：
  1. 通用 PDF 阅读+AI 总结工具；
  2. 文献管理工具（偏收藏和检索）；
  3. 通用聊天产品外挂文档问答。
- 本项目创新点不是“单点功能新奇”，而是**围绕单篇论文的闭环编排**：
  - 从上传解析到阅读、问答、导图、笔记的“统一资产模型”；
  - citation + anchor 让回答可追溯、可回跳、可复核；
  - 以学习场景为核心，不只是“问一次就结束”的聊天体验。
- 相比同类产品的优势：
  1. 结构化中间层（`parsed_json`）带来更稳定的跨模块复用；
  2. 引用与回跳链路打通，可信度高于纯生成答案；
  3. 按 Spec 渐进交付，架构更易演进到复习闭环（Anki/习题）。

一句话总结：**我们的差异化是“学习闭环 + 可追溯证据链”，而不是单纯 AI 聊天。**
