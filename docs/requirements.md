# 智能化论文学习平台需求文档

## 1. 文档目标

本文档用于沉淀智能化论文学习平台的初始需求基线，作为后续架构设计、任务拆分、接口设计和迭代开发的输入。

本文档当前处于 `v0.1` 草案阶段，重点目标是：

- 明确产品目标与业务边界
- 统一核心业务抽象
- 收敛 MVP 范围
- 标出当前假设项与待确认问题

## 2. 项目背景

论文学习通常面临以下问题：

- 论文原文阅读门槛高，结构复杂，理解成本高
- 学习资料分散，阅读、提问、笔记、复习常分布在多个工具中
- 传统 AI 总结或问答缺乏可追溯引用，学习可信度不足
- 论文学习往往停留在“读过”，难以形成可复习、可讲解、可巩固的知识资产

本项目拟构建一个围绕单篇论文的智能化学习平台。系统将每篇论文抽象为一个独立的“学习资产（Asset）”，并围绕该资产组织阅读、问答、导图、笔记、演示和复习等能力，帮助用户完成“阅读理解 -> 知识提炼 -> 互动讲解 -> 记忆复习”的完整闭环。

## 3. 产品目标

平台需要满足以下核心目标：

- 以论文 PDF 为输入源，抽象形成统一的学习资产
- 围绕学习资产构建统一工作区，集中承载论文相关附属学习资源
- 以“原文阅读 + AI 助教问答 + 思维导图”作为默认基础学习能力
- 将“演示文稿、Anki 卡组、课后习题”等能力设计为按需生成的增强能力
- 通过资产级知识库与 RAG 能力增强 AI 助教，提升回答准确性并提供可追溯引用
- 支持用户围绕具体文本片段或知识点进行锚点式笔记与复习

## 3.1 当前已确认的技术与集成约束

以下内容已由项目发起人明确确认，可视为当前阶段的架构输入：

- PDF 解析链路采用 `MinerU` 开放 API
- 由于 `MinerU` 需要可公网访问的 PDF 地址，用户上传的 PDF 需要先上传到阿里云 `OSS`
- 后续模型调用尽量统一采用阿里云百炼平台提供的 API
- 前端采用 `Vue 3` 路线
- 后端采用 `Python` 路线
- 数据库采用 `PostgreSQL`
- 向量存储采用 `pgvector`
- 基础设施优先采用 `Docker` 方式组织，便于本地开发、迁移和部署
- MVP 阶段可以先实现相对简单的 Web 界面，以便尽早验证主链路并修正问题
- Anki 首期只要求支持 `CSV` 导出，后续再扩展 `.apkg`
- 首期采用单用户开发模式推进
- 完整项目需要支持完整登录体系，并确保不同用户的图书馆、资产和衍生数据相互隔离
- 项目开发过程中默认保持“代码注释使用中文”的工程习惯

## 4. 目标用户与使用场景

### 4.1 目标用户

- 需要精读论文的学生、研究者和工程师
- 希望系统化理解经典论文的自学用户
- 需要将论文转化为讲解材料、复习材料的学习者

### 4.2 核心使用场景

- 用户上传一篇论文 PDF，快速进入可阅读、可提问、可做笔记的学习工作区
- 用户在阅读器中选中某段文字，向 AI 助教提问并获得带引用的回答
- 用户通过思维导图快速理解论文结构，并跳转到对应原文位置
- 用户围绕具体段落、知识点记录锚点笔记，后续按资产复习
- 用户在需要讲解或复盘时，按需生成互动式演示文稿、Anki 卡组和课后习题

## 5. 核心业务抽象

### 5.1 学习资产（Asset）

学习资产是平台的核心业务对象，用于统一表示一篇论文及其全部派生学习资源。

每个学习资产至少包含：

- 原始论文 PDF
- 解析后的中间结构数据
- 资产级知识库
- 原文阅读器上下文
- AI 助教问答上下文
- 思维导图
- 锚点笔记
- 演示文稿（按需）
- Anki 卡组（按需）
- 课后习题（按需）

### 5.2 资产来源

学习资产支持两种来源：

- 用户从平台预设经典论文库中选择
- 用户上传本地 PDF

### 5.3 个人图书馆

每个用户拥有自己的个人图书馆，用于管理全部学习资产。

图书馆中的每个资产类似一本书，用户点击后进入该论文的专属学习工作区。

## 6. 业务流程

### 6.1 资产创建主流程

1. 用户选择预设论文或上传 PDF
2. 系统保存原始 PDF
3. 系统执行基础解析，将 PDF 转换为统一中间层
4. 系统创建学习资产
5. 系统自动构建基础能力：
   - 原文阅读能力
   - 资产级知识库
   - AI 助教问答能力
   - 思维导图
6. 用户进入该资产工作区进行学习
7. 用户有额外需求时，按需生成增强资源：
   - 互动式演示文稿
   - Anki 卡组
   - 课后习题

### 6.2 资源生成策略

平台采用“基础能力默认生成 + 增强能力按需生成”的模式，而非“上传后自动生成全部资源”。

默认生成的基础能力：

- 文档基础解析
- 资产级知识库
- 原文阅读
- AI 助教问答
- 思维导图

按需生成的增强能力：

- 互动式演示文稿
- Anki 卡组
- 课后习题

设计目标：

- 降低首次进入工作区的等待时间
- 控制模型调用与资源生成成本
- 提升系统整体响应效率
- 让生成行为更贴近用户真实需求

## 7. 功能需求

### 7.1 资产创建与管理

#### 7.1.1 创建资产

系统应支持以下两种方式创建资产：

- 从平台预设论文库中选取论文
- 上传本地 PDF 文件

#### 7.1.2 资产基础信息

每个资产至少保存以下信息：

- 资产标题
- 作者
- 摘要
- 来源类型（系统预设 / 用户上传）
- 创建时间
- 原始 PDF 文件地址
- 解析结果地址
- 各附属资源生成状态

#### 7.1.3 图书馆管理

用户可在个人图书馆中：

- 查看资产列表
- 浏览资产基础信息
- 进入资产详情页 / 学习工作区

后续扩展：

- 删除资产
- 搜索与筛选资产

### 7.2 文档解析与中间层

#### 7.2.1 当前解析方案

系统当前采用 `MinerU` 开放 API 执行基础解析，并将解析结果沉淀为统一中间层供后续任务复用。

建议链路为：

`用户上传 PDF / 选择预设论文 -> PDF 存储到 OSS -> 生成公网可访问地址 -> 调用 MinerU 解析 -> 产出 Markdown / JSON 中间层 -> 入库并触发后续索引与资源构建`

当前中间层输出形式仍以 `PDF -> Markdown + 结构化 JSON` 为主。

#### 7.2.2 中间层作用

解析后的中间层需要服务于以下场景：

- 构建资产级知识库
- 支持 AI 助教 RAG
- 生成思维导图
- 生成互动式演示文稿
- 生成课后习题
- 生成 Anki 卡组
- 支持后续知识点关联

#### 7.2.3 解析内容要求

解析输出应尽量保留或识别以下结构：

- 标题
- 摘要
- 章节层级
- 正文段落
- 图表标题与说明
- 图片引用信息
- 参考文献
- 页码映射关系
- 段落定位信息

特别要求：

由于后续需要实现“原文引用定位”“跳转至原文”“锚点笔记挂接”“问答引用页码与段落”等能力，因此解析结果必须尽可能保留页码、段落、章节之间的映射关系。

额外要求：

- 需要保留 `MinerU` 原始返回结果与平台规范化后的中间层，便于后续调试与重跑
- 需要记录解析任务所使用的版本信息，避免未来因解析策略变化导致历史资产行为不一致

### 7.3 原文阅读器

#### 7.3.1 总体目标

平台需要提供内嵌式论文阅读器，而不是直接调用浏览器默认 PDF 打开方式。当前阶段以保留原始 PDF 阅读体验为主，优先采用 PDF.js 路线。

#### 7.3.2 当前技术倾向

用户倾向保留基于 PDF.js 的原始 PDF 阅读体验，因为其更贴近论文原貌。

阅读器同时需要为以下扩展能力预留接口：

- 文本选中提问
- 锚点笔记挂接
- 知识点联动
- 翻译功能扩展

#### 7.3.3 基础能力

阅读器至少应支持：

- PDF 页面展示
- 页面切换
- 缩放
- 文本选中
- 当前页定位

建议能力：

- 高亮选区
- 原文片段引用定位

#### 7.3.4 后续扩展方向

阅读器需为以下能力预留扩展空间：

- 段落级翻译
- 原文与结构化内容联动
- 选区问答
- 笔记挂接
- 知识点反向定位

### 7.4 AI 助教问答

#### 7.4.1 能力目标

AI 助教围绕当前论文提供问答支持，其回答应以该论文内容为核心依据，可适度补充背景知识，但不得脱离论文主体。

#### 7.4.2 回答边界

AI 助教回答原则：

- 优先基于当前论文内容回答
- 可适度补充通用背景知识，帮助用户理解论文
- 不应将无依据的模型常识当作论文结论输出
- 当问题超出论文内容时，应明确区分“论文内容”与“背景补充”

#### 7.4.3 问答输入来源

AI 助教可接收以下上下文：

- 用户手动输入的问题
- 用户在阅读器中选中的原文片段
- 当前资产的知识库检索结果
- 当前定位的页面 / 段落 / 章节信息

#### 7.4.4 引用与可追溯性

AI 助教回答时，应尽量给出清晰来源引用，包括：

- 引用页码
- 引用段落或章节
- 检索到的原文片段

### 7.5 资产级知识库与 RAG

#### 7.5.1 总体目标

系统需要为每个资产构建专属知识库，用于增强 AI 助教问答能力。

#### 7.5.2 数据来源

知识库主要来源包括：

- PDF 解析得到的正文内容
- 章节结构信息
- 图表说明
- 段落定位信息

后续可扩展来源：

- 用户笔记
- 问答记录
- 用户高亮内容

#### 7.5.3 功能要求

知识库需要支持：

- 文本切分
- 向量化索引
- 语义检索
- 检索结果与原文位置映射
- RAG 上下文注入

#### 7.5.4 检索结果要求

检索返回结果需包含：

- 片段内容
- 页码
- 段落标识
- 章节信息
- 原文跳转能力

### 7.6 思维导图

#### 7.6.1 总体目标

系统需基于论文结构与核心知识点自动生成思维导图，帮助用户建立论文整体认知框架。

#### 7.6.2 默认生成策略

思维导图属于基础能力，在资产创建后应自动生成，并作为用户初次进入学习工作区时可直接访问的资源之一。

#### 7.6.3 交互能力

思维导图不能仅为静态图片，用户应能够：

- 点击某个节点
- 跳转到对应原文位置
- 跳转到对应知识点或解释内容
- 与原文阅读、问答形成联动

### 7.7 锚点笔记

#### 7.7.1 总体目标

用户在学习过程中应能够围绕具体文本片段或知识点记录电子笔记，用于后续复习和整理。

#### 7.7.2 笔记类型

平台采用锚点笔记模式，而不是无结构笔记。每条笔记应至少关联以下对象之一：

- 某段原文文本
- 某个知识点
- 某个思维导图节点

后续可扩展：

- 关联某页演示文稿

#### 7.7.3 笔记能力

用户至少应支持：

- 创建笔记
- 编辑笔记
- 删除笔记
- 查看与某段文本或知识点相关的笔记
- 在复习时回看笔记

#### 7.7.4 复习导向

笔记模块需面向复习场景设计，因此应支持：

- 按资产查看
- 按时间排序
- 按知识点查看
- 与原文片段联动回溯

### 7.8 互动式演示文稿

#### 7.8.1 总体目标

系统应支持根据论文内容生成互动式演示文稿，用于更生动地讲解论文内容。

#### 7.8.2 互动性定义

本项目中的互动性主要指：

- 页面元素能够分步、分布动画出现
- 在讲解相应内容时可插入论文原始图片
- 演示内容具备讲解节奏与视觉引导能力

#### 7.8.3 内容要求

每份演示文稿应具备：

- 按论文结构组织的页面内容
- 每页对应讲解词
- 必要时插入论文中的原始图片
- 页面元素的渐进式展示动画

#### 7.8.4 播放能力

演示文稿需要支持：

- TTS 播报每页讲解词
- 当前页朗读完成后自动翻页
- 手动翻页
- 播放 / 暂停
- 查看当前页讲稿

#### 7.8.5 生成策略

演示文稿不属于默认自动生成内容。仅当用户提出需求时，系统才触发相应工具链生成。

#### 7.8.6 数量限制

每个资产在任意时刻最多只能拥有一份演示文稿。用户可以：

- 新建演示文稿（当当前不存在时）
- 删除已有演示文稿
- 删除后重新生成新的演示文稿

### 7.9 Anki 卡组

#### 7.9.1 总体目标

系统应基于论文核心知识点自动生成用于记忆复习的 Anki 卡组。

#### 7.9.2 输出形式

Anki 功能至少支持：

- 卡片预览
- CSV 导出

首期不要求 `.apkg` 导出，但数据模型和资源类型应为后续扩展预留空间。

#### 7.9.3 内容范围

生成内容应覆盖：

- 核心概念
- 关键术语
- 方法流程
- 重要结论
- 实验设计与结果

#### 7.9.4 生成策略

Anki 卡组为按需生成资源，不在资产创建时自动生成。

#### 7.9.5 数量限制

每个资产在任意时刻最多只能拥有一个 Anki 卡组。用户可以：

- 创建卡组
- 删除卡组
- 删除后重新生成

### 7.10 课后习题

#### 7.10.1 总体目标

系统需支持根据论文内容自动生成课后习题，帮助用户检验理解效果。

#### 7.10.2 题目范围

习题可围绕以下维度生成：

- 核心概念理解
- 方法原理分析
- 实验设计理解
- 优缺点总结
- 应用场景拓展

#### 7.10.3 生成策略

课后习题属于按需生成资源，不在资产创建时自动生成。

#### 7.10.4 数量限制

每个资产在任意时刻最多只能拥有一套课后习题。用户可以：

- 生成习题
- 删除习题
- 删除后重新生成新的习题

### 7.11 资产详情页 / 学习工作区

#### 7.11.1 总体目标

用户点击图书馆中的某个资产后，应进入该资产的专属学习工作区。

#### 7.11.2 默认能力

用户首次进入工作区时，系统至少应已具备以下能力：

- 原文阅读器
- AI 助教问答
- 思维导图
- 资产级知识库支撑

#### 7.11.3 内容结构

学习工作区统一承载以下内容：

- 论文原文阅读
- 助教问答区
- 思维导图区 / 入口
- 锚点笔记能力
- 演示文稿入口（若已生成）
- Anki 卡组入口（若已生成）
- 课后习题入口（若已生成）
- 各附属资源状态展示

## 8. 资源生命周期规则

### 8.1 基础资源

基础资源包括：

- 文档解析
- 知识库
- 阅读器
- AI 助教
- 思维导图

这些资源在资产创建时自动初始化。

### 8.2 增强资源

增强资源包括：

- 演示文稿
- Anki 卡组
- 课后习题

这些资源由用户主动触发生成。

### 8.3 数量约束

对于演示文稿、Anki 卡组、课后习题三类增强资源：

- 每类资源每个资产同一时间最多只允许存在一个
- 用户可删除
- 删除后可重新生成新的版本

### 8.4 状态管理

每类资源需具备独立状态，例如：

- 未生成
- 生成中
- 生成成功
- 生成失败
- 已删除

## 9. 非功能需求

### 9.1 用户体验

- 首次进入资产工作区时，不应因全量资源生成而长时间阻塞
- 基础能力应优先可用
- 增强能力采用按需生成策略

### 9.2 可扩展性

系统应为以下能力预留扩展空间：

- 翻译功能
- 原文与结构化内容双视图
- 多语言讲稿与 TTS
- 多资产联合检索
- 基于笔记和高亮的知识增强

### 9.3 可追溯性

AI 助教回答与知识库检索结果必须尽量可追溯到原文位置，包括页码、段落、章节与原文片段。

### 9.4 数据隔离

用户上传的资产、笔记、问答内容及衍生资源必须与用户账号绑定，确保个人数据隔离。

### 9.5 成本控制

增强资源生成采用按需触发模式，以降低大模型调用、TTS 合成和衍生内容生成成本。

## 10. MVP 范围

### 10.1 首期必做

- 用户上传 PDF / 选择预设论文
- 资产创建与图书馆展示
- PDF 基础解析
- 基于 PDF.js 优化的原文阅读器

- 资产级知识库
- AI 助教问答
- 回答引用页码 / 段落 / 原文片段

### 10.3 第三阶段

### 10.4 后续扩展

## 11. 假设项

当前为便于架构设计，先采用以下假设：

- 首期以 Web 平台为主，不考虑原生移动端
- 首期允许部分 AI 生成任务采用异步方式完成，并通过状态轮询或推送反馈
- 预设论文库的论文文件及元数据由平台预先维护，不在首期支持后台运营系统
- 首期以论文英文 PDF 为主，但产品层面预留多语言扩展空间
- 首期允许通过阿里云 OSS 暴露带时效的公网访问地址，以满足 MinerU 解析前置要求

## 12. 待确认问题

以下问题会直接影响架构和工程边界，建议后续逐步确认：

- 引用定位的最小粒度希望做到页级、段落级，还是句子级
- 思维导图是否需要支持用户手动编辑，还是首期只读
- 问答记录是否需要持久化并作为知识库增量来源
- 演示文稿最终输出是 Web 演示页面、可导出 HTML，还是两者都要
- 首期是否只接入 `MinerU` 单一解析链路，还是需要预留第二解析策略用于兜底
- 公网访问的 PDF 地址采用带时效签名 URL，还是采用受控公开路径
- 首期本地开发和部署是否都以 Docker Compose 为标准方式

## 13. 需求理解摘要

本项目的本质不是“论文 PDF 阅读器”，而是“围绕单篇论文构建完整学习闭环的智能学习工作区”。系统核心不在于单点 AI 能力，而在于将论文抽象为统一的学习资产，并让阅读、问答、知识库、思维导图、笔记和复习资源围绕该资产协同工作。

从工程落地角度看，首期最关键的是先建立可靠的资产主链路：

`PDF 输入 -> OSS 存储 -> MinerU 解析 -> 解析中间层 -> 阅读器 -> 资产级检索 -> 带引用问答 -> 导图 -> 锚点笔记`

只要这条主链路稳住，后续演示文稿、Anki、习题等增强能力都可以作为附属异步资源平滑接入。
